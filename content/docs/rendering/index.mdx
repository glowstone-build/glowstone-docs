Glowstone is built on **Spectre**, its custom GPU rendering engine. Spectre draws every frame through a real-time HDR pipeline: lit surfaces and volumetric beams accumulate in high dynamic range, then are exposed, bloomed, and tonemapped to a displayable image. The live viewport and the offline still render walk the **same path**.

## How a frame is built

Every frame — viewport or render — runs the same ordered passes. Earlier passes work in linear HDR (`Rgba16Float`); only the final tonemap step produces a displayable 8-bit image (`Rgba8Unorm`).

<Steps>
<Step>
### Forward opaque pass
Lit meshes (the set, fixtures, floor) are drawn in linear light to an HDR target.
</Step>
<Step>
### Volumetric raymarch
Spectre marches each light beam through the [environment fog box](/docs/scene/environment) at half resolution, sampling haze density and the phase function, then composites the result over the opaque layer with transmittance blending.
</Step>
<Step>
### Temporal resolve
The volumetric layer is reprojected and EMA-blended across frames to suppress banding and noise.
</Step>
<Step>
### Bloom
Bright pixels (the over-`1.0` HDR energy) are extracted by a bright pass, then blurred by a separable horizontal/vertical Gaussian at half resolution so beam cores and highlights glow.
</Step>
<Step>
### Tonemap & exposure
Exposure, the Narkowicz ACES filmic tonemap, and an sRGB encode map the HDR image (plus bloom) down to the LDR target.
</Step>
</Steps>

<Callout type="info" title="Why HDR is required">
Beam glow only reads correctly because the image is accumulated in HDR before tonemapping. Clipping to LDR too early throws away the forward-scattered radiance that makes a beam look like bright light in haze.
</Callout>

## Render matches preview

The viewport and the still render are not two separate renderers. They share the raster + volumetric path **and** the Color Management settings (`exposure`, `bloom`, `beam_intensity`, `gobo_sharpness`, `chroma_haze`), so adjusting the look updates the preview instantly and the final render inherits it.

<Callout type="success" title="WYSIWYG by design">
Frame, expose, and grade the shot in the viewport. The render reproduces that look at full resolution with higher sample counts — no surprises between preview and output.
</Callout>

The render does differ from the live preview in a few deliberate ways: it pins `render_scale` to `1.0` (full resolution), uses higher volumetric `steps` and sample counts, lets temporal accumulation converge over many frames, and drops the grid and gizmos for a clean plate. See [Rendering a Still Image](/docs/rendering/render-image) for the workflow.

## Viewport modes

A segmented **Display Mode** toggle in the viewport header switches how the 3D scene is drawn. This affects the preview only — the render always uses the full Beauty path.

Switch to `Wireframe` to check spatial layout and rigging positions, then back to `Beauty` to judge the look in haze.
</Callout>

## Output formats

The offline [render job](/docs/rendering/render-image) can write the finished plate to disk in three formats. All are post-tonemap LDR — even OpenEXR stores the tonemapped result, not raw linear HDR.

<TypeTable
  type={{
    PNG: { description: '8-bit RGBA, lossless. The default.', type: 'image', default: 'default' },
    JPEG: { description: '8-bit RGB, lossy. No alpha channel.', type: 'image', default: '—' },
    OpenEXR: { description: '32-bit float RGBA, but post-tonemap LDR — intended for float-pipeline compositors.', type: 'image', default: '—' },
  }}
/>

<Callout type="warn" title="Not yet functional">
Several render controls are visible but not yet implemented: **Render Animation** (the disabled Animation button), the **Motion Blur** and **Caustics** stub panels, and the **New Window** display mode (only the Render Tab and Keep Interface modes work). These are greyed out or have no effect.
</Callout>

## Common questions

<Accordions type="single">

  <Accordion title="Why is the per-pixel raymarch the default instead of the froxel grid?">
    The optional froxel volumetric grid is off by default because it produces blocky "fog circle" artifacts near the camera and mushy gobos. Per-pixel raymarch is cleaner and sharper for typical stage shots. It is an opt-in performance tier for very large rigs. See [Volumetric Beams & Haze](/docs/rendering/volumetrics).
  </Accordion>
  <Accordion title="Where do I change exposure and bloom?">
    In the shared Color Management controls, available both in the [Preferences](/docs/interface/preferences) panel and in Render Properties (select the World root node in the Inspector). Both edit the same values.
  </Accordion>
</Accordions>
