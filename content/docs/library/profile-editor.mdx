## Opening the editor

<Steps>
<Step>
### Select the fixture
Click the fixture in the 3D viewport or the [Scene Outliner](/docs/scene/outliner).
</Step>
<Step>
### Open the profile
With a GDTF fixture selected, choose **Fixture ▸ Edit Profile…**, or click the **Profile…** button in the fixture's [Inspector](/docs/interface) header.
</Step>
</Steps>

<Callout type="info" title="Read-only by design">
Almost everything in the editor comes straight from the GDTF file and is read-only. Only a fixture's **instance name**, **Fixture ID** (MVR fixtures only), **DMX mode** and **Move speed** are editable per placement — the GDTF profile itself is never modified.
</Callout>

The editor is organised into seven tabs.

  <Card icon={<Fingerprint />} title="Identity">Names and provenance.</Card>

## The GDTF control model

GDTF describes a fixture's controls as a small hierarchy. The Profile editor exposes all of it, tab by tab.

<Cards>
  <Card icon={<Layers />} title="DMX mode">
    A control configuration with a fixed channel count (footprint), a set of emitters and an optical wheel chain.
  </Card>
  <Card icon={<SlidersHorizontal />} title="Channel">
    One attribute (Pan, Dimmer, Color1…) occupying one or more DMX slots in the mode.
  </Card>
  <Card icon={<Hash />} title="Channel function">
    A DMX sub-range within a channel, mapped to a physical range or a wheel slot.
  </Card>
</Cards>

A fixture's mode lives at the top. Each mode contains channels; each channel contains one or more channel functions.

## Identity

The Identity tab shows who made the fixture and how this placement is labelled. The descriptive fields are read-only; the instance-specific fields can be edited. A fixture's GDTF **Description**, when present, is shown in a separate section below the grid.

### Modelled movement

Below the optical chain a section marked **MODELED — NOT IN GDTF** carries the synthesized movement values, each with a *modeled* badge. The GDTF carries no real degrees-per-second figure, so Glowstone derives one so moving fixtures feel realistic.

### Emitter selector

A column on the left, headed **EMITTERS**, lists the emitters (light sources) in the current mode.

- For multi-emitter fixtures, each emitter is listed by index and name. Click one to see its individual beam optics.
- Single-source fixtures show **(single source)**.

<Callout type="info">
The number of emitters depends on the selected DMX mode. Switching modes on the [DMX Modes](#dmx-modes) tab can change which emitters appear here.
</Callout>

## DMX modes

The DMX Modes tab lists every DMX mode the fixture type supports. A mode is a named control layout: a moving head might offer `Standard` (16 channels), `Extended` (32 channels) and a `Low Resolution` variant. The mode determines:

- **Footprint** — total DMX slots the fixture consumes.
- **Channels** — which attributes are addressable, and at what resolution.
- **Emitters** — how many light sources the fixture drives in this mode.
- **Optical chain** — the colour, gobo, prism, animation and frost wheels exposed.

Each row shows the mode name, its channel count (footprint) and its emitter count.

<TypeTable
  type={{
    'Mode name': { description: 'The mode label.', type: 'e.g. Standard, Extended' },
    'Channel count': { description: 'Number of DMX slots the mode occupies (footprint).', type: 'e.g. 16 ch' },
    'Emitter count': { description: 'Number of light sources active in this mode.', type: 'e.g. 4 emitters' },
  }}
/>

**Click a mode to switch the fixture to it.** This is a per-instance setting: it reconfigures which channels are available and updates the geometry, emitter and wheel assignments. The fixture stays placed in the scene — only its control layout changes.

<Callout type="warn">
Changing the mode changes the channel footprint. If the fixture is already patched, review its address afterwards so it does not overlap a neighbour — the address range it occupies may have grown or shrunk. See [Patching Fixtures](/docs/dmx/patching).
</Callout>

## Channels

The Channels tab is a table of every DMX channel in the current mode, with the mode name and footprint shown above it. Each channel binds one **attribute** to one or more DMX slots — Pan, Tilt, Dimmer and Color1 are all channels. Rows are **resolved** — for multi-emitter or instanced geometries, each instance gets its own DMX address.

| Column | Meaning |
| --- | --- |
| `Addr` | DMX slot number within the fixture footprint (1-based). Shows `—` for a virtual channel with no footprint. |
| `Geometry` | The geometric component the channel controls — the reference-instance name when instanced, e.g. `Pan`, `Tilt`, `Lens2`. |
| `Attribute` | The control attribute name, e.g. `Pan`, `Tilt`, `Dimmer`, `Color1`. |
| `Function` | The primary channel-function name, e.g. `Pan`, `Focus`, `Dimmer`. |

### 8-bit and 16-bit channels

A single logical channel may occupy more than one slot. 16-bit attributes such as `Pan` and `Tilt` use a coarse and a fine slot, so the channel count can exceed the number of distinct attributes.

  type={{

    function: { description: "Human-readable name of the channel's primary range, e.g. \"Pan\", \"Focus\", \"Dimmer\".", type: 'text' },
    geometry: { description: 'The geometric component this channel drives, e.g. "Pan Axis", "Base Fixture", "Zone2".', type: 'text' },

To map attributes to live beam behaviour, see the [Beam Parameter Reference](/docs/fixtures/beam-engine).

### Channel functions

A single channel rarely does just one thing. **Channel functions** carve the channel's DMX range into sub-ranges, each mapped to a physical range or a wheel slot. On a colour wheel channel, one band of DMX values selects red, another selects blue, and a high band might spin the wheel.

<TypeTable
  type={{
    name: { description: 'Function name, human-readable, e.g. "Pan", "Open", "Red".', type: 'text' },
    attribute: { description: 'Attribute this function belongs to, e.g. "Pan".', type: 'text' },
    dmx_from: { description: 'DMX value where this sub-range starts, normalised 0–1 over the full channel.', type: '0–1' },
    physical_from: { description: 'Physical range start, e.g. -270 for pan degrees.', type: 'number' },
    physical_to: { description: 'Physical range end, e.g. 270 for pan degrees.', type: 'number' },
    wheel: { description: 'Linked wheel name for wheel-based functions, e.g. "Colour1".', type: 'text | none' },
  }}
/>

For wheel channels, each function can carry a list of **channel sets** — the exact `DMX → wheel-slot` mappings from the GDTF. These drive the slot names shown in the inspector and the swatches/gobo thumbnails on the [Wheels](#wheels) tab.

<Accordions type="single">
  <Accordion title="What is a channel set?">
    A channel set is one GDTF `<ChannelSet>` row: a DMX start value (normalised 0–1) paired with a wheel slot index (1-based; 0 means no slot link). It is how a discrete DMX value such as "value 32 = Gobo 1" is recorded.
  </Accordion>
  <Accordion title="Why does my channel show physical ranges in degrees?">
    Continuous attributes like Pan and Tilt map their DMX range to a physical range — for example -270° to +270° of pan travel. The channel function's physical_from / physical_to define that mapping.
  </Accordion>
  <Accordion title="Where do gobo and prism controls come from?">
    They come from the mode's optical chain rather than from a plain physical range. Wheel controls split into Value (slot selection or insertion %), Index (indexed rotation of the chosen slot) and Spin (continuous rotation, where 0.5 = stop). See [Gobos, Animation, Prism & Frost](/docs/fixtures/gobos-prism).
  </Accordion>
</Accordions>

### Instanced cells

When a fixture uses `GeometryReference` instancing — for example a multi-zone LED bar — each instanced cell gets its **own DMX address**. The Channels tab shows these as resolved rows: the same attribute repeats once per cell, each at a distinct slot.

## Wheels

The Wheels tab previews every optical wheel defined in the GDTF — colour, gobo, prism, animation and frost — with a thumbnail for each slot. Fixtures with no wheels show **No wheels.**

<ImagePlaceholder
  label="Screenshot"
  ratio="16 / 9"
  caption="The Wheels tab of the Profile editor for a moving head, showing two wheels stacked vertically: a Gobo1 wheel with a row of PNG gobo image thumbnails (Open, plus patterned gobos), and a Colour1 wheel rendered as a row of solid colour swatches (Open/white, red, blue, CTO amber). Each wheel header shows its name and slot count, e.g. 'Gobo1 · 8 slots'. The cursor hovers a gobo slot showing its name tooltip."
/>

Each wheel header shows its **name** (e.g. `Colour1`, `Gobo1`, `Prism1`) and **slot count** (e.g. `8 slots`). For every slot:

- If slot media is available (a gobo image or animation frame), the PNG thumbnail is shown.
- Otherwise a colour swatch is drawn from the GDTF CIE xyY colour data, converted to RGB.
- Hover a slot to see its name (e.g. `Open`, `Red`, `CTO`).

For how these wheels drive the rendered beam, see [Gobos, Animation, Prism & Frost](/docs/fixtures/gobos-prism).

## Defaults

The Defaults tab shows the DMX default value for each channel in the current mode, taken from the GDTF `InitialFunction`. These are the values the fixture holds before any console data arrives — useful for confirming that, say, a shutter sits open and a colour wheel sits on `Open` at rest.

<TypeTable
  type={{

<Callout type="info">
These defaults define each fixture's starting look until live values arrive over [Art-Net or sACN](/docs/dmx/connectivity).

## Frequently asked

<Accordions type="single">
  <Accordion title="Can I edit a fixture's beam angle or colour temperature?">
    The Optics values shown come from the GDTF and are read-only. The editable per-instance settings are the instance name, Fixture ID (MVR fixtures only), DMX mode and the modelled Move speed.
  </Accordion>
  <Accordion title="Why does the channel count change when I switch modes?">
    Each DMX mode is a distinct control configuration with its own channel footprint and emitter set. Switching modes reconfigures the channel map, so the totals on the Channels and Defaults tabs change to match.
  </Accordion>
  <Accordion title="A wheel slot shows a flat colour instead of a gobo. Why?">
    The GDTF for that slot carries no media image, so Glowstone draws a swatch from the slot's CIE colour data instead. Slots with embedded PNG media show the image.
  </Accordion>
</Accordions>
