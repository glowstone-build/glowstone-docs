Glowstone reads and writes one native project format, one scene-interchange format, plus the fixture, geometry, and image formats those archives bundle. This page lists each one and where it is used.

## At a glance

| Format | Type | Read | Write | Notes |
| --- | --- | --- | --- | --- |
| `.glow` | Project | Yes | Yes | Native project (self-describing MessagePack). Bundles everything inline; exact round-trip; opens across app versions. |
| `.mvr` | Scene | Yes | Yes | ZIP scene-interchange archive. Round-trips with grandMA3, Vectorworks, Capture, Depence. |
| `.gdtf` | Fixture definition | Yes | No | Read-only. Imported into the library and bundled into `.glow`/`.mvr`. |
| `.glb` / `.3ds` | Geometry | Yes (via MVR) | Yes (via MVR) | Stage/truss/prop meshes. Carried inside MVR archives, never as loose files. |
| PNG / JPEG | Textures | Yes | Yes | Fixture wheels and MVR/screen content. Carried verbatim in archives. |
| PNG / JPEG / OpenEXR | Render output | — | Yes | Saved still-image output. |
| HDR | HDRI environment | Yes | — | Sky/ambient environment map, bundled inline into `.glow`. |

<Callout type="info">
`.gdtf` files and `.glb`/`.3ds` geometry are never loaded or kept as loose files in a project. They are bundled inside the `.glow` project (and re-emitted into `.mvr` on export), so a saved show is fully self-contained.
</Callout>

### `.glow` — the native project

A single binary file that captures the entire show. Use it for everyday work: it is what `File ▸ Save` (`Cmd`/`Ctrl`+`S`) and `File ▸ Open Project…` read and write. Because all referenced assets ride along inside the file, a `.glow` project opens with no external dependencies.

The body is a single self-describing [MessagePack](https://msgpack.org/) dump of the same serde types the app runs on, so it captures the exact in-memory state rather than a lossy interchange (that is MVR's job). Because structs are written as field-name maps, each value is matched by **name** on load — so a project saved by any version opens in any other: fields a build doesn't have take their defaults, and fields it doesn't recognise are ignored. Pyro devices and per-fixture sequence numbers are part of the same body.

<Callout type="info" title="Crash-safe saves">
Saves are atomic: Glowstone writes a temporary sibling file first, then renames it onto the target.
</Callout>

<Callout type="success" title="Opens across versions">
The format is self-describing and **not** version-gated. A project from an older build opens in a newer one (new fields take their defaults); a project from a newer build opens best-effort in an older one (unrecognised fields are ignored). The leading version stamp is a diagnostic only — it never refuses a file on version alone. The earlier `.archie` format used a different binary encoding and is not read by current builds.
</Callout>

### `.mvr` — scene exchange

A ZIP archive of a scene description plus its referenced resources. Use it to bring a rig in from, or send one out to, another tool. See [MVR Scene Exchange](/docs/projects/mvr) for the full import/export workflow and round-trip details.

## Bundled resource formats

These formats are read and written only as part of a `.glow` or `.mvr` archive. You do not manage them as loose files in an open project.

### `.gdtf` — fixture definitions (read-only)

Glowstone reads GDTF but never writes it. Import one with `File ▸ Import GDTF Fixture…`; the definition is added to the library and a fixture instance is placed in the scene. On save, the original `.gdtf` archive bytes are bundled into the project (one copy per unique fixture type) and re-parsed on open. See [Importing GDTF Fixtures](/docs/library/import-gdtf).

### `.glb` / `.3ds` — geometry

Mesh formats for static stage objects: trusses, decks, set pieces, and props. Glowstone imports and exports these only through MVR archives. Each model keeps a per-model transform matrix from import (used for unit conversion), so geometry comes in at the correct scale and round-trips faithfully.

### PNG / JPEG — textures

Used for fixture gobo and colour wheels and for MVR screen content. Inside MVR archives they are carried verbatim so they re-bundle unchanged on export.

<Callout type="warn">
LED screen image content is bundled inline (its decoded bytes ride along, like the HDRI) in the native `.glow` project, so it survives save and reopen. On **MVR** round-trip, however, only the image file name is carried — the bytes are not — so a re-imported screen shows no image until you re-pick it. Live sources (NDI, CITP, pixel-map DMX) round-trip by source identifier only. See [Screen Content Sources](/docs/media/content-sources).
</Callout>

## Image formats for rendering

<Cards>
  <Card icon={<Image />} title="HDR — environment input" href="/docs/scene/world">
    An HDRI map drives sky lighting and ambient light. Its bytes are bundled inline into the project.
  </Card>
  <Card icon={<FileDown />} title="PNG / JPEG / OpenEXR — render output" href="/docs/rendering/render-image">
    Choose PNG (lossless 8-bit RGBA), JPEG (small, no alpha), or OpenEXR (32-bit float container).
  </Card>
</Cards>

When you render a still image, the output format is part of the render settings: `PNG`, `JPEG`, or `OpenEXR`. The EXR readback is the post-tonemap LDR plate written into a float container — convenient for a float-pipeline compositor, but not raw linear HDR for highlight recovery. See [Rendering a Still Image](/docs/rendering/render-image).
