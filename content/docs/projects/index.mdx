A Glowstone project is a single self-contained `.glow` file (magic `GLOW\0`). It bundles the entire show inline — scene, fixtures, GDTF definitions, geometry, render settings, preferences, DMX patch/config, selection groups, cues and pyro devices — so a project is portable with no external assets to track. The format is **self-describing**, so a project saved by any version of Glowstone opens in any other (see [compatibility](#backward--forward-compatible)).

## Project basics

Use the **File** menu (or the welcome screen) to manage projects. The window title shows the current project name; an untitled project has no on-disk path until you save it.

## The welcome screen

The welcome splash appears at startup, and on demand from **Window ▸ Welcome** (choosing **File ▸ New Project** also re-opens it). It offers:

- **New Project** — clears the current scene (fixtures, geometry, screens, groups, cues) and opens an untitled project.
- **Open…** — a native file picker filtered to `.glow` files.
- **Recover Last Session** — reopens the auto-saved session from the last run. Shown only when a valid autosave exists (see [Crash recovery](#autosave--crash-recovery)).
- **Recent Files** — the most recent projects (up to 8 shown here), newest first.

Use **Continue without a project** to dismiss the splash and keep the current scene.

## Creating, opening and saving

<Steps>
<Step>
### Start a new project

Choose **File ▸ New Project** (or **Window ▸ Welcome**), then click **New Project** on the welcome screen. This clears the current scene and opens an untitled project with no on-disk path.
</Step>
<Step>
### Open an existing project

Choose **File ▸ Open Project…** and pick a `.glow` file. Glowstone checks the file magic and decodes it, restoring the scene, fixtures (with their sequence numbers), GDTF library, camera, render settings, preferences, DMX patch/config, selection groups, cues and pyro devices.
</Step>
<Step>
### Save your work

Press `Cmd`/`Ctrl`+`S` (**File ▸ Save**). If the project already has a path, it writes there. If it is untitled, a Save As dialog opens first.
</Step>
<Step>
### Save a copy

Choose **File ▸ Save As…** (`Cmd`/`Ctrl`+`Shift`+`S`) to write to a new location. The save dialog forces the `.glow` extension; the default name is `show.glow` for an untitled project, or the current filename otherwise. This updates the window title and adds the path to Recent Files.
</Step>
</Steps>

### File operation shortcuts

| Shortcut | Action |
| --- | --- |
| `Cmd`/`Ctrl`+`S` | Save (Save As if untitled) |
| `Cmd`/`Ctrl`+`Shift`+`S` | Save As… |

<Callout type="info" title="Atomic, crash-safe writes">
Saving writes to a temporary sibling file first, then renames it onto the target. If a save is interrupted, your existing project file is left intact — there are no partial or corrupt saves.
</Callout>

## What the .glow file preserves

Everything the show needs lives inside the one file. GDTF definitions, geometry meshes, textures and the HDRI are all stored inline — no external references to break.

<Callout type="idea" title="Projects are portable">
Because GDTF files are embedded in the `.glow` file, you can hand a project to anyone and it opens with the correct fixtures — no need to share or re-install `.gdtf` files alongside it. Each fixture's GDTF is re-parsed from the bundled bytes on open.
</Callout>

### File layout

The `.glow` file is a short magic + version stamp followed by a single self-describing [MessagePack](https://msgpack.org/) body. Structs are written as field-name maps, so on load each value is matched by **name** rather than byte position — which is what makes the format compatible across versions (see below). Everything is in the one body; there is no separate trailer.

      <File name="Bundled GDTF archive bytes + per-fixture spec keys" />
      <File name="Geometry meshes, textures, HDRI (inline in Scene)" />
      <File name="Camera, render settings, preferences" />
      <File name="DMX patch + config, selection groups, cues" />
    </Folder>
  </Folder>
</Files>

<Callout type="success" title="Backward & forward compatible">
Because values are matched by field name, your shows keep opening across updates. A project from an **older** Glowstone opens in a newer build — fields it never had simply take their defaults. A project from a **newer** Glowstone opens **best-effort** in an older build — fields the old build doesn't recognise are ignored. The version stamp is a diagnostic only (it sharpens the error message in the rare case a newer file fails to decode); it never refuses a file on version alone.
</Callout>

<Callout type="info">
The only changes that aren't automatically compatible are renaming a field, changing a field's type, or removing an enum variant — those need a deliberate migration in the app.
</Callout>

## Autosave & crash recovery

Glowstone writes a periodic background autosave (roughly every 20 s, when the scene has content) so you can recover after an unexpected quit.

- The autosave is `last-session.glow` in Glowstone's [cache directory](#where-glowstone-stores-files) — on macOS, `~/Library/Caches/build.glowstone.glowstone/`.
- After a crash, if a valid autosave exists, the welcome screen shows a **Recover Last Session** button. A recovered session opens as untitled, so the next Save prompts for a destination instead of overwriting the autosave.

<Callout type="warn">
Autosave is a safety net, not a substitute for saving. Save your project with `Cmd`/`Ctrl`+`S` regularly so your named file stays current.
</Callout>

## Recent projects

The Recent Files list (in the **File ▸ Open Recent** submenu and on the welcome screen) shows recently opened or saved projects, newest first.

- The stored list holds up to 12 entries, deduplicated by path; the welcome screen shows up to 8.
- Projects that no longer exist on disk are pruned automatically, so the list only shows files you can actually open.

## Where Glowstone stores files

Apart from your `.glow` projects, Glowstone keeps a few per-user files — your keymap, workspaces, camera bookmarks, Library favourites, recent-files list, and the GDTF Share cache — in the standard OS locations (derived from the `build.glowstone.glowstone` app identifier).

<Tabs items={['macOS', 'Linux', 'Windows']}>
  <Tab value="macOS">
    - **Config / Data**: `~/Library/Application Support/build.glowstone.glowstone`
    - **Cache**: `~/Library/Caches/build.glowstone.glowstone`
  </Tab>
  <Tab value="Linux">
    - **Config**: `~/.config/glowstone`
    - **Cache**: `~/.cache/glowstone`
    - **Data**: `~/.local/share/glowstone`
  </Tab>
  <Tab value="Windows">
    - **Config**: `%APPDATA%\glowstone\glowstone\config`
    - **Cache**: `%LOCALAPPDATA%\glowstone\glowstone\cache`
    - **Data**: `%APPDATA%\glowstone\glowstone\data`
  </Tab>
</Tabs>

| File / folder | Directory | Holds |
| --- | --- | --- |
| `keymap.json` | Config | Keyboard rebinds |
| `workspaces.json` | Config | Saved panel layouts |
| `bookmarks.json` | Config | Camera bookmarks |
| `library.json` | Config | Library Recents & Favourites |
| `inspector.json` | Config | Inspector category open/closed state |
| `recent.json` | Config | Recent-projects list |
| `share-credentials.json` | Config | GDTF Share login (only with **Remember me**) |
| `share-list.json` | Cache | Cached GDTF Share catalogue |
| `last-session.glow` | Cache | Autosave / crash recovery |
| `gdtf/` | Data | Downloaded GDTF Share fixtures |

<Callout type="info">
On macOS, Config and Data resolve to the same Application Support folder. If the OS directories can't be resolved, downloaded GDTFs fall back to a temporary `glowstone-gdtf` folder.
</Callout>

## Frequently asked

<Accordions type="single">
  <Accordion title="Do I need to keep the original .gdtf files after importing?">
    No. Once a fixture is in your project, its GDTF is bundled into the `.glow` file and re-parsed on open. The original `.gdtf` is not required to reopen the project.
  </Accordion>
  <Accordion title="Why are identical truss copies still fast after reload?">
    On load, Glowstone re-deduplicates identical geometry blobs by their resource name, restoring the memory sharing from import time so repeated meshes draw as instanced copies rather than separate objects.
  </Accordion>
  <Accordion title="What about old .archie files?">
    The native format is now `.glow`. It uses a different, self-describing encoding (MessagePack) from the old `.archie` binary, so the Open dialog accepts `.glow` only and earlier `.archie` files are not loaded by current builds.
  </Accordion>
  <Accordion title="Can I open a project saved by a newer version of Glowstone?">
    Yes, best-effort. The format is self-describing and **not** version-gated: a newer file's recognised fields load, and anything the older build doesn't know is ignored. The version stamp only sharpens the error message in the rare case decoding actually fails.
  </Accordion>
</Accordions>
