A Glowstone scene is a flat collection of placed things — fixtures, static objects, LED screens, pyro devices and fog volumes — plus a single world that lights it all. The Scene Outliner shows a tidy hierarchy on top of that flat list, but underneath every entity sits in one place.

## The scene model

Glowstone stores the scene as flat `Vec`s of plain data — no nesting in the storage. The tree you see in the outliner is a view derived from those lists every frame.

| Kind | What it is | Examples |
| --- | --- | --- |
| **Fixtures** | Controllable light sources | Moving heads, PAR cans, lasers |
| **Objects** | Static imported geometry | Stage decks, truss, set pieces |
| **Screens** | LED video walls and panels | A 4×2 cabinet wall |
| **Pyro** | Placed pyro devices | CO2 jets, cold-spark machines |
| **Environments** | Fog / haze volumes | A `Fog Box` that makes beams visible |
| **World** | A single global sky + ambient | One HDRI per scene |

<Callout type="info">
There is exactly one **World** per scene. Fixtures, objects, screens, pyro devices and environments can each have any number of instances.
</Callout>

Pyro is a first-class placed entity with its own inline DMX patch (not the fixture patch table). It is documented on its own page — see [Pyro Devices](/docs/fixtures/pyro).

## Stable entity IDs

Every fixture, object, screen, pyro device and environment carries a session-stable identity (`EntityId`). The outliner uses these IDs to track which rows are expanded and where a selection range is anchored, so that state survives even when you add, delete or reorder entities.

<Callout type="idea">
Because identity is tracked by ID and not by row position, your expanded groups and selection anchors stay put as the scene changes around them. The ID is a runtime concept (`serde`-skipped, reassigned on load) and is not written into the `.glow` file.
</Callout>

## The unified object handle

Fixtures, geometry, screens, pyro devices and environments differ in almost every way, but Glowstone treats them identically for a handful of common operations. A single unified handle (`ObjectRef`) lets all these kinds share one path for:

- **Transforming** — move, rotate and scale (see [Moving, Rotating & Scaling](/docs/scene/transforming))
- **Duplicating** — copy any mix of objects at once (see [Duplicate & Array](/docs/scene/duplicate-array))
- **Framing** — every object reports a world-space bounding box so the camera can frame it
- **Re-selecting** — after a duplicate, the copies are selected for you

This is why the same gizmos and shortcuts work whether you grabbed a moving head, a truss, an LED wall, a CO2 jet, or a fog box. (World is not a placed object and is excluded from this handle.)

## One active kind at a time

Selection keeps a multi-select list for **one** kind at a time — fixtures *or* objects *or* screens *or* pyro. The first item in the active list is the *primary* selection that drives single-object editing. World and Environment are single-only selections that are mutually exclusive with everything else.

<Callout type="info" title="Switching kinds">
A plain click, or a `Cmd`/`Ctrl`-click on an item of a different kind, switches the active kind. Selecting the World or an environment clears any fixture, object, screen or pyro selection. See [Selecting Objects](/docs/scene/selection) for the full set of rules.
</Callout>

## How the outliner derives its tree

The outliner rebuilds its hierarchy from the flat scene on every frame. Conceptually the tree looks like this:

<Files>
  <Folder name="Scene" defaultOpen>
    <Folder name="World" defaultOpen>
      <Folder name="Environment">
        <File name="Fog Box 1" />
      </Folder>
    </Folder>
    <Folder name="Fixtures" defaultOpen>
      <File name="PAR Can 1" />
      <File name="Moving Head 1" />
    </Folder>
    <Folder name="Objects">
      <File name="Stage Deck" />
      <File name="Truss" />
    </Folder>
    <Folder name="Screens">
      <File name="LED Wall 1" />
    </Folder>
    <Folder name="Pyro">
      <File name="CO2 Jet 1" />
    </Folder>
  </Folder>
</Files>

Each row shows just its name (the type icon conveys the kind); the secondary string — a group's item count, or a leaf's type/profile — moves to the row's hover tooltip. Fixture rows additionally show a leading **sequence number** and, on the right, a channel-footprint chip (`Nch`) and the patch address. The full breakdown of rows, filters, search and sort lives on the [Scene Outliner](/docs/scene/outliner) page.
